02 / Lifecycle

One incident, start to finish

The exact sequence exercised in the live smoke test: first failure, remediation, a re-failure inside the cooldown window, escalation, and eventual recovery. 

sequenceDiagram
    participant J as Jenkins Master
    participant O as Observer (Go)
    participant R as Redis
    participant T as Teams
    participant W as Remediation Worker (Python)
    participant A as AWS SSM

    O->>J: GET /computer/api/json?depth=2
    J-->>O: agent list (offline: true, reason, labels)
    O->>R: SETNX alert:dedup:{fingerprint} EX 300
    R-->>O: first-seen
    O->>T: POST Adaptive Card — "Agent Offline"
    O->>R: LPUSH remediation:jobs
    W->>R: BLPOP remediation:jobs
    W->>R: EXISTS circuit:tripped / circuit:cooldown
    R-->>W: no active cooldown
    W->>A: SendCommand self_heal.sh (or dry-run log)
    W->>R: SET circuit:cooldown EX 1800
    W->>T: POST Adaptive Card — "Remediation Triggered"
    Note over J,A: node fails again before cooldown expires
    O->>R: SETNX alert:dedup (fresh incident window)
    W->>R: EXISTS circuit:cooldown
    R-->>W: still active — re-failure confirmed
    W->>R: SET circuit:tripped (no TTL)
    W->>T: POST Adaptive Card — "Requires SRE Intervention"
    Note over O: later poll — node is back online
    O->>T: POST Adaptive Card — "Recovered"
03 / Breakdown

Input → process → output → result

Every stage read the same way: what goes in, what the code decides, what gets sent out, what changes in the real world.

PHASE 1Detect & Notify
read-only
Input

Jenkins Remote Access API response, polled every 30s per master.

displayName, offline, offlineCauseReason, assignedLabels[]

Process
  • Fingerprint = SHA256(masterURL+node+reason)
  • SETNX in Redis, 300s window
  • Re-alert only at counts 5 / 10 / 50 / 100
  • Resolve Teams webhook by first matching label
Output

Adaptive Card v1.5 POST to the team's Teams webhook — master, node, cause, timestamp, occurrence count, a link to the agent's status page.

Result

Ops sees the outage within one poll cycle. Repeat polls of the same incident stay silent instead of paging the same channel every 30 seconds.

PHASE 2Self-Heal
destructive · dry-run by default
Input

One job per brand-new incident, popped from remediation:jobs:

{master, node, instance_id, reason, labels}

Process
  • Circuit breaker check first (see below)
  • DryRunExecutor logs the action only
  • AWSSSMExecutor runs self_heal.sh for real via SSM
  • Runbook: prune docker, clear workspace, restart agent service
Output

Teams card — "Remediation Triggered" — plus a circuit:cooldown key set for 30 minutes (production default).

Result

The node heals with no human touching a keyboard — or, in local/dry-run mode, you see exactly what would have run before it ever touches AWS.

PHASE 2bCircuit Breaker
safety net
Input

A new incident on a node whose circuit:cooldown key is still active — i.e. it broke again right after being "fixed."

Process
  • Trip: SET circuit:tripped (no TTL)
  • Observer reads this key and stops enqueueing new jobs for the node
  • circuit:escalated (1h TTL) stops repeat pages
Output

Teams card — "Requires SRE Intervention" — sent once, not on every subsequent poll.

Result

Auto-healing is disabled for that node until a human clears it. No infinite fix-loop chewing through AWS API calls on a genuinely broken host.

PHASE 3Observe
read-only
Input

Prometheus scrapes two targets every 5s: the observer's own /metrics, and a Jenkins-metrics endpoint.

Process
  • Grafana datasource + dashboard auto-provisioned on boot
  • Zero manual click-through setup
  • 5 panels, thresholds baked in (see below)
Output

A live dashboard: Agent Fleet Status, Queue Backlog, Memory, Disk, and Observer Health (alerts sent / suppressed / poll errors).

Result

One glance answers "is the fleet healthy right now" — no digging through Teams history or SSH-ing into hosts.

04 / State

The circuit breaker, as states

The same logic as the sequence above, drawn as a state machine — the one path that doesn't self-clear.

stateDiagram-v2
    [*] --> Healthy
    Healthy --> Remediating: agent offline (first-seen incident)
    Remediating --> Cooldown: remediation executed, 30 min grace window starts
    Cooldown --> Healthy: agent recovers before window ends
    Cooldown --> Tripped: agent fails again inside the window
    Tripped --> Healthy: SRE clears manually — redis-cli DEL circuit:tripped:...
    Tripped --> Tripped: further offline polls — alerts continue, no re-remediation

NOTE

What's real vs. mocked in this build: the poll → dedup → Teams path is fully real. Remediation defaults to DryRunExecutor — it never touches AWS unless REMEDIATION_MODE=aws-ssm is set with real credentials, and even then it's only unit-tested against a mocked boto3 client, not a live account.

The Jenkins-shaped Prometheus metrics come from internal/promfake, a stand-in for a real Jenkins Prometheus plugin — and the memory/disk panels stand in for a real host exporter like node_exporter, which a production deployment still needs to add per agent.
