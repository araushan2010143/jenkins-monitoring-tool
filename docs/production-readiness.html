Tier A

Configuration only

The code already has a slot for this. Fastest wins — fill it with something real instead of a mock.

01Real Jenkins connection
Currentcmd/mockjenkins fixture server
→
NeededA service account with read access to /computer/api/json
Where
cp configs/masters.example.json configs/masters.json — fill in real url, username, apiToken.
How
Manage Jenkins → Security — the account needs Overall/Read and Agent/Read at minimum. Generate the API token from that account's config page.
CHECKConfirm curl -u user:token https://jenkins/computer/api/json?depth=2 works before wiring it in — some setups gate this behind CSRF crumbs even on GET.
02Real Teams routing
Currentwebhook-sink echo server
→
NeededOne Teams incoming webhook per destination channel
Where
cp configs/routing.example.json configs/routing.json — fill in real webhook URLs.
How
Per channel in Teams → Workflows app → "Post to a channel when a webhook request is received" → copy the generated URL.
CHECKPOST one real card manually first. Teams' actual Adaptive Card v1.5 rendering has quirks the spec doesn't capture, especially on mobile — confirm it looks right before trusting it in production.

Currentconfigs/instances.local.json — one fake ID
→

Where
cp configs/instances.example.json configs/instances.json

Tier B

Real infrastructure has to exist first

No amount of local configuration substitutes for these — the real thing has to be stood up.

04AWS account + IAM for remediation
CurrentNo explicit credential wiring — relies on boto3's default chain
→
NeededScoped IAM policy + SSM Agent running on every target instance
Where
remediation/executor.py (AWSSSMExecutor) — no code change needed if using an IAM role; the credential just has to exist wherever the container runs.
How
  • IAM policy scoped to ssm:SendCommand + ssm:GetCommandInvocation, restricted by tag or instance ARN — not ssm:*
  • Verify the SSM Agent is actually running on target hosts
  • Decide: IAM role attached to the container (best practice) vs. AWS_ACCESS_KEY_ID/AWS_SECRET_ACCESS_KEY env vars (simpler, worse)
SAFETYFlip REMEDIATION_MODE=aws-ssm against a disposable test EC2 instance first — never a real Jenkins agent on the first try.
05Validate self_heal.sh against real agent hosts
CurrentPorted from the blueprint, never run against a real host
→
NeededConfirmed match against your actual agent AMI/provisioning
Where
remediation/scripts/self_heal.sh
How
It hardcodes /home/jenkins/workspace, the jenkins-agent.service unit name, a required JENKINS_AGENT_JNLP_URL env var, and ~/.gradle/~/.m2 cache paths. Diff every one of these against your real hosts.
RISKA mismatched path or service name means this silently no-ops on the wrong target, or fails loudly and never restarts the agent — instead of fixing anything.
06Real Jenkins Prometheus plugin + a real host exporter
Currentinternal/promfake fabricates every metric
→
NeededThe real plugin, plus node_exporter on every agent host
Where
observability/prometheus.yml — replace the jenkins job's target with your real Jenkins /prometheus/ path; add one scrape job per agent's node_exporter:9100.
How
Install the plugin, confirm curl https://jenkins/prometheus/ returns real metrics, then compare its actual metric names against observability/grafana/dashboards/jenkins-fleet.json's queries.
EXPECT EDITSThe dashboard's queries were written against the blueprint's documented names and validated only against the mock. Plugin versions have changed metric naming before — expect to edit targets[].expr. The memory/disk panels specifically need rewriting to node_memory_MemAvailable_bytes / node_filesystem_free_bytes, not the promfake names.
Tier C

Needs actual building

Genuine gaps — nothing to configure your way around, this is unwritten code or infrastructure.

07Persistent storage for Redis, Prometheus, Grafana
CurrentAll three are stateless containers
→
NeededNamed volumes, or a managed Redis
Where
docker-compose.yml — add redis-data, prometheus-data, grafana-data volumes; redis-server --appendonly yes for Redis specifically.
WHY IT MATTERSA Redis restart today loses dedup state and circuit breaker trips at the same time — every "requires SRE intervention" node quietly resets to healthy. For real production, prefer a managed Redis (e.g. ElastiCache) over a container.
08Secrets management
CurrentGitignored local JSON files on disk
→
NeededOrchestrator-native secrets, or a real secrets manager
Where
internal/config/config.go (LoadMasters / LoadRouting / LoadInstances) and remediation/config.py (load_routing) — both only read from disk today.
How
  • Mount from Kubernetes Secrets / ECS task secrets as files at the same paths — no code change needed
  • Or add a real Secrets Manager / Vault fetch — this is an actual new loader function, alongside the existing file-based ones
09Monitoring the monitor
CurrentNothing watches the observer itself
→
NeededA dead-man's-switch alert independent of the observer
Where
New observability/alertmanager.yml + a Prometheus rules file referenced from observability/prometheus.yml.
How
A rule like absent(jenkins_observer_polls_total) for 2m, routed through Alertmanager (or a dumb external uptime check on /healthz) to a channel that doesn't depend on the observer being alive.
WHY IT MATTERSIf the observer process crashes, right now nothing tells anyone — the whole alerting pipeline goes silent with it.
10Scale validation
CurrentTested against one mock master, one or two agents
→
NeededConfirmed behavior at real fleet size
Where
internal/poller (recovery tracking uses a Redis SCAN per master per poll) and HTTP_TIMEOUT (10s default) in internal/config.
How
  • Confirm the SCAN pattern holds up at your actual agent count
  • Test multiple real masters polled concurrently — supported in code, never run for real
  • Raise HTTP_TIMEOUT if depth=2 is slow on a controller with hundreds of agents
11A real deployment target
Currentdocker-compose.yml — one host, dev-oriented
→
NeededKubernetes manifests, or Terraform for ECS/EC2 — depends on the target
Where
New — doesn't exist yet in any form.
DECIDE FIRST"Add Kubernetes support" and "add Terraform for ECS" are different pieces of work. Worth deciding the actual target before building either.

Suggested order

Each step is safe to ship on its own — nothing here requires the next step to already be done.

1

Tier A — real Jenkins + real Teams

Read-only, zero risk. Gets you a genuinely useful monitoring tool with no destructive capability yet.

2

Prometheus plugin + node_exporter

Extends real visibility. Still zero risk.

3

Persistence + secrets management

Makes the above durable and safe to actually deploy.

4

AWS SSM against a disposable test instance

Prove remediation works before it can touch anything real.

5

Flip to real remediation on one low-stakes agent

Not the whole fleet on day one.

6

Monitoring-the-monitor + scale validation

Before trusting this at full fleet size.

7

Deployment target

Kubernetes, Terraform, or whatever fits — once it's clear what this is actually deploying onto.
