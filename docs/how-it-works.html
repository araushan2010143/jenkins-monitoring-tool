In Plain English

What this system actually does — and why

No jargon. Think of your Jenkins build servers as patients, and this system as a small hospital that watches over them: something checks on them constantly, tells the team the moment one gets sick, tries an automatic treatment, and calls in a human if the treatment doesn't hold.

The Big Picture

Six things happen, in order

Every single incident — from the smallest hiccup to a full outage — follows this same path.

👀

Watch

Every 30 seconds, check whether any build server has gone quiet.

🔍

Notice

If one's down, figure out why — and make sure we haven't already said something about it recently.

📣

Tell someone

Send a message straight to the right team's channel in Microsoft Teams.

🛠️

Try to fix it

Attempt an automatic repair — no human needs to log in.

🚨

Call for help, if needed

If the same server breaks again right after being "fixed," stop retrying and ask a person instead.

📊

Show the big picture

A live screen shows which servers are healthy right now, at a glance.

Meet the Parts

Seven pieces, one job each

For each one: what it is, why it exists, when it acts, and how it does its job — no code, no commands.

🖥️
The Patients
Jenkins build servers ("agents")
What
The actual computers that run your software's builds and tests.
Why it matters
They're the whole reason this exists — when one goes down, work backs up behind it.
When they act up
Any time — out of memory, out of disk space, lost network, crashed process.
How we relate to them
We never touch them directly at first — we only watch, from the outside, through Jenkins.
🩺
The Nurse on Rounds
The Observer — cmd/observer
What
A small program that never sleeps, checking on every server on a fixed schedule.
Why it exists
Nobody wants to manually refresh a status page every 30 seconds, all day, forever.
When it acts
Every 30 seconds, for every server it's told to watch.
How it works
It asks Jenkins "who isn't answering right now?" — and for anyone who isn't, it kicks off everything else on this page.
📓
The Notebook
Redis (shared memory)
What
A fast, shared notebook that every other part can read from and write to.
Why it exists
Without it, the Nurse would repeat "server's down!" every 30 seconds forever — nobody wants 120 messages an hour about the same problem.
When it's checked
Every single time something happens, to ask "have we already dealt with this?"
How it works
Like sticky notes with timers — "already reported this 5 minutes ago," "already tried fixing this once" — that quietly remove themselves when the timer runs out.
💬
The Paging System
Microsoft Teams
What
Where the actual humans find out something's wrong.
Why it exists
A problem nobody's told about isn't really "monitored" — someone has to see it to act on it.
When it fires
The first time a server goes down; again if it's still down after a while (so it never goes silent); and again the moment it recovers.
How it works
A neatly formatted card lands in the right team's channel, with what's wrong and a link to check it directly.
🧰
The On-Call Repair Crew
Remediation Worker — remediation/worker.py
What
A separate program whose only job is attempting fixes.
Why it exists
Most outages have a boring, repeatable fix — clear disk space, restart a stuck service. A human shouldn't have to do that by hand at 3am.
When it acts
Right after a brand-new problem is reported — but only if nobody's tried fixing this exact server recently.
How it works
Right now it runs in "dry run" mode — it writes down exactly what it would do, without touching anything real, so it's fully safe to test. One setting flips it to actually do it.
🔒
The Safety Switch
Circuit Breaker — remediation/circuit_breaker.py
What
A rule that says "don't keep trying the same fix if it isn't working."
Why it exists
Imagine a server that breaks, gets auto-fixed, breaks again two minutes later, gets auto-fixed again, breaks again — forever. That wastes time and can make things worse.
When it trips
Only when a server breaks again shortly after being "fixed." A brand-new problem on a healthy server never trips it.
How it works
It flips a switch that says "stop auto-fixing this one" and pages a human instead — and it stays flipped until a person turns it back off, on purpose.
📊
The Big Screen
Grafana Dashboard
What
A live webpage showing the health of every server at a glance.
Why it exists
Sometimes you just want to look at one screen and know if everything's fine, without reading a wall of chat messages.
When it updates
Constantly — every 5 seconds. It's never stale.
How it works
Panels turn red, yellow, or green based on simple rules — for example, "less than 5 GB of disk space left? turn red."
A Day in the Life

One outage, start to finish

A realistic walkthrough, minute by minute — this is the exact sequence that was tested live before this was ever written down.

9:00 AM

All servers are healthy. The Big Screen is all green.

9:14 AM

ec2-agent-01 runs out of memory and disconnects from Jenkins.

9:14 AM

The Nurse notices on its very next check, within 30 seconds.

Detected

9:14 AM

It checks the Notebook — "have I said anything about this in the last 5 minutes?" No. So it speaks up.

Teams: "ec2-agent-01 is offline — out of memory"

9:14 AM

The Repair Crew picks up the job. No recent fix attempt on file, so it goes ahead — in dry-run mode, it just logs what it would do.

Teams: "Remediation Triggered (dry-run)"

9:44 AM

Thirty minutes pass with no repeat failure. The server comes back online on its own.

Teams: "Recovered"

What if it had broken again instead?

If ec2-agent-01 had failed a second time within that 30-minute window, the Safety Switch would have tripped right then: auto-fixing stops for that server, and a "Requires SRE Intervention" message goes out instead — once, not on every check afterward. A person then has to clear it by hand once it's genuinely fixed.

Common Questions

Things people usually ask

Does it ever fix something without telling anyone?
No. Every single action — noticing a problem, attempting a fix, giving up and calling a human, recovering — posts a message. Nothing happens silently.
Can it get stuck endlessly "fixing" the same broken server?
No — that's exactly what the Safety Switch prevents. Two failures close together and it stops trying, permanently, until a person says otherwise.
Is anything fake or simulated right now?
Yes, on purpose. Real repairs on real servers, and pulling metrics from a real Jenkins install, aren't switched on yet — because there's no real Jenkins or AWS account to safely test against. Everything currently runs against realistic stand-ins, so every piece of logic above is fully proven before it's ever pointed at something real.
What if two different servers break at the same time?
They're handled completely independently. One being broken — or being fixed, or being stuck — never blocks or slows down any of the others.
What's the one thing a human actually has to do?
Clear the Safety Switch after it trips. Everything else — noticing, reporting, attempting a fix, updating the dashboard — happens on its own.
